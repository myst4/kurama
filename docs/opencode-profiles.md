# OpenCode model profiles

A **profile** in Kurama's OpenCode integration is a *named model configuration*: a
complete parallel set of agent entries in `opencode.json` that share the SDD
prompt files and differ only in their `model`. It lets you keep, say, a default
setup on one model family and a `cheap` profile on smaller models, and switch
between them inside a single OpenCode session with **Tab**.

Profiles rest entirely on three **native OpenCode primitives** — nothing else:

1. multiple `mode:primary` agents, cycled with **Tab** in the TUI;
2. the per-agent `model` field; and
3. `{file:...}` prompt references, so every profile points at the same shared
   prompt files instead of duplicating prompt text.

There is no "profile" object in OpenCode. A profile is an emergent convention over
those primitives, generated by one idempotent `jq` pass in `scripts/setup.sh`.

## What gets installed

Installing a profile named `NAME` splices these entries into
`~/.config/opencode/opencode.json` (existing agents and their user-edited models
are preserved):

- **`kurama-orchestrator`** — `mode:primary`. The profile conductor. Its `task`
  permission is scoped to `sdd-*-NAME`, so it only delegates to its own suffixed
  subagents. Prompt: `{file:./AGENTS.md}`.
- **`sdd-<phase>-NAME`** — one `mode:subagent`, `hidden` agent per SDD phase
  (`init`, `explore`, `propose`, `spec`, `design`, `tasks`, `apply`, `verify`,
  `archive`). Each references the shared prompt file
  `{file:~/.config/opencode/prompts/sdd/sdd-<phase>.md}`.

The shared prompt files under `~/.config/opencode/prompts/sdd/` are installed by
setup for both the `multi` config and any profile, so N profiles reference **one**
prompt file per phase and vary only `model`.

## Installing a profile

```sh
# Default name "kurama", models left as the <your-provider/your-model> placeholder
./scripts/setup.sh --agent opencode --opencode-mode multi --opencode-profile kurama

# Named profile with a model set for every phase in one shot
./scripts/setup.sh --agent opencode --opencode-mode multi \
  --opencode-profile cheap:anthropic/claude-haiku-4-5

# Omit the name (default "kurama") but still set the model
./scripts/setup.sh --agent opencode --opencode-profile :anthropic/claude-haiku-4-5
```

Flag grammar: `--opencode-profile NAME[:provider/model]`. `NAME` must be lowercase
alphanumeric with dashes; if omitted it defaults to `kurama`. The optional
`provider/model` sets the `model` field of every profile agent on a first install.

Interactively, `setup.sh --agent opencode` asks whether to create a named profile
after choosing the mode; answer empty to skip.

## Using a profile

The `/sdd-*` slash commands do **not** route to a profile. Every command carries
a frontmatter-pinned `agent:` field, so it always runs against a *base* agent no
matter which primary is Tab-selected:

- executor commands (`/sdd-apply`, `/sdd-verify`, `/sdd-spec`, …) route straight
  to the base `sdd-<phase>` subagents;
- meta commands (`/sdd-new`, `/sdd-continue`, `/sdd-ff`, `/sdd-status`) route to
  the base `sdd-orchestrator`.

None of them touch `kurama-orchestrator` or the `sdd-<phase>-NAME` agents, so a
`/sdd-*` command run while `kurama-orchestrator` is selected still uses the base
agent at its default model — not the profile's.

To run the SDD flow **on the profile's per-phase models**, drive it with a
freeform (non-slash) request to the selected primary:

1. Launch `opencode`.
2. Press **Tab** to cycle the `mode:primary` orchestrators until
   `kurama-orchestrator` is the selected primary.
3. Type a freeform request — e.g. `run the SDD apply phase for change X` or
   `start a new SDD change: <description>`. As the selected primary,
   `kurama-orchestrator` handles the message directly and delegates only to its
   `sdd-<phase>-NAME` subagents, each of which carries the profile's configured
   model.

## Editing models by hand

There is no model picker. To change a phase's model, edit the `model` field of the
matching `sdd-<phase>-NAME` entry in `~/.config/opencode/opencode.json`:

```json
{
  "agent": {
    "sdd-apply-cheap": { "model": "anthropic/claude-sonnet-4-5", "...": "..." }
  }
}
```

Re-running setup with the same profile name is **idempotent**: it preserves every
model you edited by hand (it captures existing `model` values before splicing and
restores them afterward), so you can re-sync prompts and structure without losing
your model choices.

## What Kurama does NOT port from gentle-ai

Kurama ships only the *runtime shape* of a profile (native multi-primary +
per-agent model + shared prompt files). The following authoring conveniences from
gentle-ai's Go binary are intentionally **not** reproduced — they have no
bash/`jq` equivalent:

- **Interactive model picker + provider/auth detection** (backed by the OpenCode
  models cache). Kurama replacement: edit `model` fields or pass `:provider/model`.
- **Reasoning-effort variants** (e.g. `gpt-5` low/medium/high) via the
  `model-variants` plugin cache. Kurama profiles are plain `provider/model` only.
- **A TUI profile manager** (create/edit/delete screens) and a
  `sync --profile` / `--profile-phase` CLI. Kurama's equivalent is the one-shot
  `--opencode-profile` flag; editing is a re-run or a hand-edit of `opencode.json`.
- **External-profile-file auto-detection** (`profiles/*.json` strategy switching).
  Kurama ships only the generated single-profile overlay.

The `background-agents.ts` delegation plugin is retained (unchanged by profiles):
its disk-persisted async `delegate` survives compaction, which is an advantage over
the native `task` tool.
